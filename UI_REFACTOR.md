# Echoscope UI Refactor Plan

Date: 2026-02-13  
Scope: UI/UX refactor without changing DSP/scan algorithms

## 1) Цель

Сделать интерфейс заметно проще, быстрее для повседневного сценария и эргономичнее на мобильных устройствах, не ломая текущую инженерную функциональность.

### Целевые UX-результаты

- Снизить когнитивную нагрузку первого экрана (минимум полей и текста в базовом режиме).
- Сократить время до первого успешного `Ping/Scan` для нового пользователя.
- Уменьшить визуальный шум в основной сессии (графики в приоритете, диагностика вторична).
- Сохранить полный экспертный контроль через прогрессивное раскрытие.

### KPI (первый релиз)

- Видимых контролов в базовом режиме: не более 12–16.
- Видимых `details[open]` по умолчанию: 1–2 (сейчас 7).
- Количество длинных explanatory-блоков на первом экране: сократить минимум на 60%.
- Время до `Init → Ping`: сократить минимум на 30% в ручном UX smoke-тесте.

## 2) Базовый аудит текущего `index.html`

Текущая поверхность UI (по разметке):

- `input`: 63
- `select`: 8
- `button`: 15
- `textarea`: 1
- `details`: 13 (из них `open`: 7)
- `canvas`: 9
- `inline style`: 103 вхождения
- `hint`-абзацы: 25
- `label`: 72

### Ключевые проблемы

1. **Смешаны разные уровни задач в одном потоке**
  - Операционный сценарий (`Init/Ping/Scan`) соседствует с редкими экспертными DSP-параметрами и диагностикой (`Device Info`, `Network/Peers`, `log`).

2. **Слишком длинный control-first поток**
  - На узких экранах визуализации идут после длинной панели контролов, что ухудшает “видимость результата” и обратную связь.

3. **Перегрузка объясняющим текстом**
  - Длинные “About mode” и подробные описания каждого графика одновременно видимы/легко раскрываются.

4. **Недостаточная визуальная иерархия**
  - Много равнозначных полей, мало явного приоритета “главное действие vs тонкая настройка”.

5. **Слабая maintainability разметки**
  - 103 inline-style усложняют единообразие и безопасный рефакторинг.

---

## 3) UX-принципы рефактора

1. **Progressive disclosure**: сначала только то, что нужно для запуска.
2. **Action-first**: главные действия всегда сверху и доступны.
3. **Result-first**: визуализация и обратная связь видимы как можно раньше.
4. **One mental model per zone**: каждая зона решает один тип задач.
5. **No power loss**: экспертные функции остаются, но не мешают базовой работе.
6. **ID compatibility first**: существующие `id` не удаляются в MVP-итерации.

## 4) Целевая информационная архитектура

## 4.1 Главные зоны

1. **Базовый режим (по умолчанию)**
  - `Init/Ping/Scan/Stop`
  - `Mode`
  - `Angle`
  - `Device preset`
  - `Max range`
  - `Scan step`
  - `Direction axis`
  - `Show trace`

2. **Экспертный режим**
  - Все advanced scan/calibration/virtual-array/blanking/tracking controls
  - Параметры probe-режимов (chirp/mls/golay/multiplex)

3. **Диагностика**
  - `Device Info`
  - `Network / Peers`
  - `log`
  - Расширенная sanity-информация

## 4.2 Визуализации

- По умолчанию открыта только одна ключевая визуализация (Heatmap).
- `Range profile` и `Geometry` доступны через компактные `details`.
- Управление colormap/dB оставить рядом с Heatmap (как контекстный control).

## 5) Предлагаемая структура интерфейса (MVP)

## 5.1 Верхняя панель действий

- Sticky row: `Init`, `Ping`, `Scan`, `Stop`, `Status`.
- `Ping/Scan` всегда рядом с короткой обратной связью (`bestReadout`, `dirReadout`).

## 5.2 Базовые настройки

- Отдельный компактный блок “Quick setup”.
- Все вычисляемые подсказки (`computedC`, `computedListenMs`, `computedScanDwell`) показывать кратко, без длинных пояснений.

## 5.3 Экспертный блок

- Один общий `details`: “Expert settings”.
- Внутри — подподразделы:
  - Probe advanced
  - Scan quality advanced
  - Calibration advanced
  - Tracking visualization
  - SAFT / Virtual array

## 5.4 Диагностика

- Один `details`: “Diagnostics”.
- Внутри: `Device Info`, `Peers`, `log`, расширенные калибровочные детали.
- По умолчанию закрыт.

## 5.5 Текст и микро-копирайтинг

- Длинные “About ... mode” заменить на:
  - 1 строку “Для чего режим”
  - кнопку/ссылку “Подробнее”
- Описания графиков сократить до 1–2 предложений.

---

## 6) Mobile-first эргономика

1. **Порядок секций на узком экране**
  - Сначала action bar и ключевая визуализация (Heatmap), затем контролы.

2. **Крупные touch targets**
  - Кнопки и ключевые переключатели: комфортный hit-area.

3. **Снижение прокрутки**
  - Спрятать редкие настройки в закрытые `details`.

4. **Sticky action bar**
  - Чтобы `Ping/Scan/Stop` не терялись при прокрутке вниз.

---

## 7) Техническая стратегия рефактора

Ограничение MVP: меняем layout/структуру и UX-поток, но не меняем вычислительный pipeline.

## 7.1 Жёсткие инварианты совместимости

- Не удалять существующие `id`, которые используются в `src/ui/app.ts`, `src/ui/controls.ts`, `src/ui/readouts.ts`.
- Разрешено перемещать блоки по DOM и скрывать их через `details`/CSS.
- Новые wrapper-контейнеры добавлять без изменения существующих идентификаторов контролов.

## 7.2 Файлы MVP-итерации

- `index.html` — новая иерархия секций и progressive disclosure.
- `src/styles/main.css` — убрать inline-style в утилитарные классы, настроить mobile-поток.
- `src/ui/app.ts` — минимальные обработчики для нового UI-переключателя режимов (если добавится).

## 8) Детальный поэтапный план

## Phase 0 — Подготовка и safety net

Цель: зафиксировать baseline и снизить риск поломки wiring.

Задачи:
- Составить список всех используемых `id` (источник истины для совместимости).
- Добавить smoke-checklist ручной проверки основных действий.
- Подготовить feature toggle для включения нового layout (опционально).

Артефакт:
- Чеклист UI wiring + baseline screenshots (desktop/mobile).

Критерий готовности:
- Любое перемещение секций не ломает `Init/Ping/Scan/Calibrate`.

## Phase 1 — IA перестройка (`index.html`)

Цель: разделить базовый, экспертный и диагностический сценарии.

Задачи:
- Выделить `Quick setup` блок из текущего верхнего control-слоя.
- Объединить advanced секции в `Expert settings`.
- Объединить `Device Info`, `Network/Peers`, `log` в `Diagnostics`.
- Сократить default-open секции визуализаций до 1–2.

Критерий готовности:
- Пользователь видит минимально достаточный набор контролов без прокрутки в типовом desktop viewport.
- На mobile ключевые действия и первая визуализация доступны раньше.

## Phase 2 — CSS нормализация (`src/styles/main.css`)

Цель: убрать хаотичные inline-style и выстроить визуальную иерархию.

Задачи:
- Вынести повторяющиеся inline-style в классы (`stack`, `panel`, `inline-help`, `compact-row`, `actions-sticky`, и т.п.).
- Настроить layout для mobile-first последовательности.
- Добавить стили для clear hierarchy: главные controls заметнее, advanced — спокойнее.

Критерий готовности:
- Существенно меньше inline-style в `index.html`.
- Визуальная структура читается с первого скролла.

## Phase 3 — UX-поведение и микро-копирайтинг

Цель: уменьшить когнитивный шум и ускорить освоение.

Задачи:
- Сжать длинные подсказки в короткие формулировки.
- Для длинных объяснений добавить отдельное раскрытие “Подробнее”.
- Проверить, что shortcuts (`I/P/S/Esc/C`) остаются валидными и видимыми.

Критерий готовности:
- На первом экране нет длинных “стен текста”.
- Быстрый сценарий читается как линейный поток: Init → Ping → Scan.

## Phase 4 — Валидация и rollout

Цель: безопасно ввести новый UI без регрессий.

Задачи:
- Ручной smoke по ключевым сценариям (desktop + mobile).
- Регрессионная проверка переключения режимов probe и calibration.
- A/B наблюдение (если есть возможность): скорость выполнения базовых задач.

Критерий готовности:
- Нет функциональных регрессий.
- Достигнуты KPI по сокращению видимой сложности.

## 10) Критерии приёмки (Definition of Done)

UI считается успешно рефакторнутым, если одновременно выполнено:

1. Базовый сценарий доступен без перехода в экспертные секции.
2. Все текущие `id`-привязки продолжают работать.
3. Количество видимых контролов и текстовых блоков в default-state заметно снижено.
4. Визуализация и status/readouts находятся в зоне раннего внимания пользователя.
5. Mobile UX не требует длинного прокручивания до первого полезного результата.

---

## 11) Риски и mitigation

1. **Риск**: поломка wiring из-за перестановки DOM.  
   **Митигатор**: не менять `id`, проверять по чеклисту после каждого этапа.

2. **Риск**: power-user считает, что «функции пропали».  
   **Митигатор**: все advanced функции сохранить в явном `Expert settings`.

3. **Риск**: визуально легче, но функционально всё ещё сложно.  
   **Митигатор**: текстовая компрессия + приоритетные presets + быстрый поток действий.

4. **Риск**: mobile ухудшится из-за sticky/больших блоков.  
   **Митигатор**: отдельный проход теста на iPhone viewport.

## 12) План проверки после внедрения

### Ручной smoke (обязательный)

- `Init Audio` → успешный статус.
- `Ping` обновляет profile/readouts.
- `Scan` обновляет heatmap/progress/readouts.
- `Calibrate` работает, sanity/geometry корректно отображаются.
- Переключение `mode` скрывает/показывает соответствующий блок параметров.
- `Network/Peers` и `Device Info` доступны через diagnostics.

### UX smoke

- Новый пользователь может выполнить `Init → Ping → Scan` без открытия экспертных секций.
- На mobile first meaningful visual feedback появляется без длинного скролла.
